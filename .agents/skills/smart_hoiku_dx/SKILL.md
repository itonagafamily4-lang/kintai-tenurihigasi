# スマート保育DX - 勤怠管理システム開発記録

このスキルファイルは、これまでに構築された「スマート保育DX」の勤怠管理・休暇管理システムの全体像と実装の記録をまとめたものです。今後の機能追加や改修を行う際のコンテキスト（前提知識）として使用できます。

## 1. システム概要
保育園の職員（正規、時短、パートなど多彩な雇用形態）の勤怠打刻、休暇申請、および管理者によるそれらの承認・集計業務を効率化するためのWebアプリケーション。

- **スタック:** Next.js (App Router), React, TypeScript, Prisma, SQLite (Better-SQLite3), CSS Modules
- **認証/認可:** NextAuthベース（セッション管理）、`ADMIN`, `STAFF` のロール管理

## 2. 主要な機能の全体像

### 職員向け機能（STAFF）
- **ダッシュボード (Dashboard.tsx)**
  - 出勤・退勤の打刻（ボタンでワンクリック）
  - 実労働時間の自動計算
  - 残業時間、残業理由の入力
  - 短時間勤務における短縮時間（Short Time Value）の記録
  - 食事の有無（食事回数）の選択
- **休暇管理 (LeaveManagement.tsx)**
  - 休暇申請フォーム（全休、午前半休、午後半休、時間有給）
  - 特休（指定休）、感染症特休（コロナ、インフルエンザなど）の申請
  - 有休の残日数、消化率のプログレスバー表示
  - 過去の申請履歴と承認ステータス（承認待ち、承認済み、却下）の確認
- **勤怠・休暇履歴の確認 (AttendanceHistory.tsx)**
  - 月ごとのカレンダー形式やリスト形式での出退勤履歴・休暇履歴の振り返り
  - 食事の有無（○/✗表示による直感的な確認）
  - 印刷用レイアウトに最適化された1ページ出力
- **カレンダー表示 (Calendar.tsx)**
  - 自分のシフトや勤怠状況、行事予定、他の職員の休暇予定をカレンダー上で可視化

### 管理者向け機能（ADMIN - AdminPanel.tsx 傘下）
- **勤怠確認・集計**
  - 所属全職員の月別勤怠履歴・各種時間の集計表示
  - 人事・給与計算システムの締め日（例：毎月10日締め等）に応じた対象期間でのデータ集計
  - 社労士用エクスポート（CSVおよびExcel出力機能）
    - 雇用形態別（正規、時短、パート）の自動集計・グループ化出力
    - 全体のサマリー情報（総出勤日数、総残業、総食事代など）の出力
- **各種マスタ・設定の管理**
  - **職員の管理**: 職員の追加、編集、初期有休の自動付与（入職日等に基づくロジック）
  - **有休残高の手動調整**: `AdminStaffLeaveBalance.tsx` による付与日数・使用日数の個別上書き
  - **特休・指定休マスタの管理**: `AdminSpecialLeaveSettings.tsx` における慶弔休暇・夏季休暇など園独自の特別休暇ルールの追加
- **休暇承認フロー**
  - 職員が申請した休暇リクエストの一覧表示と「承認」「却下」のアクション
- **一斉付与・年度更新 (AdminAnnualLeaveGrant.tsx)**
  - 毎年4月1日等の基準日に合わせた全職員への有休一括付与
  - 付与前のプレビュー表示（対象者、勤続年数、付与予定日数、繰越日数の確認）
  - 法的要件に基づいた自動計算（比例付与テーブルの適用、産休・育休期間の出勤みなし処理）
  - 二重付与を防止するべき等性の確保（同じ年度に再実行しても利用実績や時間休取得履歴が消えない設計）
  - 2年の時効（繰越上限）に基づいた正確な残日数計算

## 3. データベーススキーマ（Prisma）
主要なモデル構造：
1. **Organization**: 施設情報、締め日の設定（例：`closingDay`=10 で前月11日から当月10日までの集計）
2. **Staff**: 職員マスタ。`employmentType` (REGULAR, SHORT_TIME, PART_TIME)、`role` (ADMIN, STAFF)、入職日、有休残高などのリレーションを持つ
3. **Attendance**: 日々の勤怠レコード。出退勤時刻、`actualWorkHours`, `overtimeHours`, `shortTimeValue`, `mealCount`, `overtimeReason` などを保持
4. **LeaveRequest**: 休暇申請レコード。`leaveType` (FULL_DAY, HALF_DAY, HOURLY, SPECIAL_SICK, SPECIAL_OTHER)、ステータス (PENDING, APPROVED, REJECTED)
5. **LeaveBalance**: 職員ごとの有休や指定休の付与・残高履歴
6. **SpecialLeaveMaster**: 園独自の特別休暇（指定休）の種類とルールの定義

## 4. 特徴的なUI/UX・フロントエンドの実装
- **CSS Modulesによるスタイリング**: 各コンポーネントごとのカプセル化（`LeaveManagement.module.css`等）
- **直感的な操作**: 
  - 食事の有無を「丸（○）」と「バツ（✗）」のテキストで分かりやすく表示（打刻していない日は「—」ハイフン）。
  - 休暇の種類に応じて理由欄の表示/非表示を動的に切り替え。特休の場合は入力不要化。
- **印刷レイアウトの堅牢化 (AttendanceHistory.module.css)**
  - 履歴印刷時に`@media print` を活用し、`zoom: 0.82` や `transform` で確実にA4縦に収まるよう自動調整。
  - セルのパディングやフォントサイズを微調整して文字の視認性を担保。
- **インテリジェントなアラート機能**
  - 打刻漏れ（出勤のみ・退勤のみ等）の自動検知とログイン時のアラート表示。
  - 有給休暇（全休・特休等）の取得日は、打刻がなくても「打刻漏れ」として扱わない法的・実務的な例外処理。

## 5. 今後の拡張を見据えた留意点
- 職員数が40名規模で1年間稼働した場合のデータ量は数十MBと非常に軽量に収まる設計になっており、SQLiteで十分にスケーラブル。
- 締め日のロジック（`getClosingPeriod`等）は、組織ごとに可変で対応できるよう構築されているため、複数組織への展開（マルチテナント）も視野に入っている。
- 特休はユーザーへの表示名がシステム上は「特休」だが、管理者側では園独自の休暇の設定などを扱うために「指定休」という名称で管理するよう意図的にUI上での名称分離を行っている（ユーザー要望による）。
- **法的整合性とデータ保全**: 有休の付与ロジックにおいて、2年間の消滅時効と繰越制限を正確に実装。また、管理操作（一斉付与の再実行など）によって、既に職員が入力・消化した休暇実績（`usedDays`や`timeLeaveUsedHours`）が消失しないよう、常に既存データを考慮して更新（upsert/update）を行う設計を徹底している。
